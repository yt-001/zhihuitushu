### 🔴 问题1：API路径冲突导致路由映射异常

**问题描述**:
`AdminController` 和 `BookManageController` 存在API路径冲突。两个Controller都定义了 `/api/admin/books` 相关的接口：

- `AdminController.java` (第28行): `@RequestMapping("/api/admin")`
  - 第91行: `@GetMapping("/books")` → 完整路径 `/api/admin/books`
  - 第120行: `@GetMapping("/books/{id}")` → 完整路径 `/api/admin/books/{id}`
  - 第132行: `@PostMapping("/books")` → 完整路径 `/api/admin/books`
  - 第143行: `@PutMapping("/books/{id}")` → 完整路径 `/api/admin/books/{id}`
  - 第158行: `@DeleteMapping("/books/{id}")` → 完整路径 `/api/admin/books/{id}`
  - 第167行: `@PostMapping("/books/batch-delete")` → 宂整路径 `/api/admin/books/batch-delete`

- `BookManageController.java` (第21行): `@RequestMapping("/api/admin/books")`
  - 第74行: `@GetMapping` → 完整路径 `/api/admin/books`
  - 第61行: `@GetMapping("/{id}")` → 完整路径 `/api/admin/books/{id}`
  - 第31行: `@PostMapping` → 完整路径 `/api/admin/books`
  - 第41行: `@PutMapping("/{id}")` → 完整路径 `/api/admin/books/{id}`
  - 第51行: `@DeleteMapping("/{id}")` → 完整路径 `/api/admin/books/{id}`
  - 第84行: `@DeleteMapping("/batch")` → 完整路径 `/api/admin/books/batch`

**影响**:
Spring Boot在启动时可能无法正确映射路由，或随机选择其中一个Controller处理请求，导致功能异常。

**建议修改方案**:
1. **方案一（推荐）**: 移除 `AdminController` 中所有图书管理相关代码（第86-175行），完全使用 `BookManageController` 处理图书管理请求。
2. **方案二**: 调整 `AdminController` 的路径，例如改为 `@RequestMapping("/api/admin/legacy")`。

**修改示例**（方案一）:

### 🔴 问题2：关键词搜索条件构建逻辑错误

**问题描述**:
`AdminController.java` 第103行的关键词搜索条件构建存在逻辑错误：

```java
wrapper.like(Book::getTitle, keyword).or().like(Book::getAuthor, keyword);
```

这会生成错误的SQL条件：
```sql
WHERE title LIKE '%keyword%' OR author LIKE '%keyword%'
```

这会导致无论其他筛选条件（categoryId、status）如何，只要keyword匹配标题或作者，记录就会被选中。

**影响**:
当用户同时使用关键词搜索和分类筛选时，分类筛选失效，返回错误的数据。

**建议修改方案**:
参考 `BookServiceImpl.java` 第147-151行的正确实现：

```java
wrapper.and(w -> w.like(Book::getTitle, queryDTO.getKeyword())
        .or()
        .like(Book::getAuthor, queryDTO.getKeyword()));
```

**修改示例**:
```java
// AdminController.java 第103行修改为：
if (keyword != null && !keyword.trim().isEmpty()) {
    wrapper.and(w -> w.like(Book::getTitle, keyword)
            .or()
            .like(Book::getAuthor, keyword));
}
```

### 🟡 问题4：N+1查询性能问题

**问题描述**:
`BookServiceImpl.java` 的 `convertToVO` 方法（第222-232行）在转换每本书籍时都会查询分类信息：

```java
private BookVO convertToVO(Book book) {
    BookVO vo = new BookVO();
    BeanUtils.copyProperties(book, vo);
    // 查询分类名称
    if (book.getCategoryId() != null) {
        Category category = categoryMapper.selectById(book.getCategoryId());
        if (category != null) {
            vo.setCategoryName(category.getName());
        }
    }
    return vo;
}
```

**影响**:
当调用 `listBooks` 分页查询时，如果返回N条书籍，会产生N+1次数据库查询（1次查书籍列表 + N次查分类），严重影响性能。

**建议修改方案**:
1. **方案一**: 在 `listBooks` 方法中批量查询所有分类，然后用Map缓存
2. **方案二**: 使用MyBatis-Plus的关联查询或JOIN查询
3. **方案三**: 在Book实体中添加 `@TableField(exist = false)` 的categoryName字段，使用SQL JOIN查询

**修改示例**（方案一）:
```java
@Override
public IPage<BookVO> listBooks(BookQueryDTO queryDTO) {
    Page<Book> pageParam = new Page<>(queryDTO.getPage(), queryDTO.getSize());
    // ... 构建wrapper ...
    
    IPage<Book> bookPage = bookMapper.selectPage(pageParam, wrapper);
    
    // 批量查询分类
    Set<Long> categoryIds = bookPage.getRecords().stream()
            .map(Book::getCategoryId)
            .filter(Objects::nonNull)
            .collect(Collectors.toSet());
    
    Map<Long, String> categoryNameMap = new HashMap<>();
    if (!categoryIds.isEmpty()) {
        List<Category> categories = categoryMapper.selectBatchIds(categoryIds);
        categoryNameMap = categories.stream()
                .collect(Collectors.toMap(Category::getId, Category::getName));
    }
    
    // 转换为VO
    IPage<BookVO> voPage = bookPage.convert(book -> {
        BookVO vo = new BookVO();
        BeanUtils.copyProperties(book, vo);
        vo.setCategoryName(categoryNameMap.get(book.getCategoryId()));
        return vo;
    });
    
    return voPage;
}
```

### 🟡 问题6：使用RuntimeException而非业务异常

**问题描述**:
多处使用 `RuntimeException` 抛出业务错误，例如：
- `AdminServiceImpl.java` 第27行: `throw new RuntimeException("用户名不存在")`
- `AdminServiceImpl.java` 第32行: `throw new RuntimeException("账号已被禁用")`
- `AdminServiceImpl.java` 第35行: `throw new RuntimeException("密码错误")`
- `BookServiceImpl.java` 第82行: `throw new RuntimeException("书籍不存在")`

**影响**:
- 无法区分不同类型的业务异常
- 前端无法获取准确的错误码和错误信息
- 异常处理不够规范

**建议修改方案**:
创建自定义业务异常类 `BusinessException`：

```java
// 新建 exception/BusinessException.java
package com.yt.zhihuitushu.exception;

import lombok.Getter;

@Getter
public class BusinessException extends RuntimeException {
    private final Integer code;
    
    public BusinessException(Integer code, String message) {
        super(message);
        this.code = code;
    }
    
    public BusinessException(String message) {
        this(400, message);
    }
}
```

修改使用处：
```java
// AdminServiceImpl.java
throw new BusinessException(401, "用户名不存在");
throw new BusinessException(403, "账号已被禁用");
throw new BusinessException(401, "密码错误");

// BookServiceImpl.java
throw new BusinessException(404, "书籍不存在");
```

### 🟡 问题7：前端分页参数名与后端不一致

**问题描述**:
前端和后端的分页参数名称不一致：

- **前端** (`Books.vue` 第96-104行):
  ```javascript
  const res = await getBookList({
    page: currentPage.value,
    pageSize: pageSize.value,  // 使用 pageSize
    keyword: searchQuery.value,
    categoryId: categoryFilter.value,
    status: statusFilter.value,
    sortBy: sortField.value,
    sortOrder: sortOrder.value === 'descending' ? 'desc' : 'asc'
  })
  ```

- **后端** (`BookQueryDTO.java` 第15-21行):
  ```java
  private Integer page = 1;
  private Integer size = 10;  // 使用 size
  ```

- **后端** (`AdminController.java` 第93-94行):
  ```java
  @RequestParam(defaultValue = "1") Integer page,
  @RequestParam(defaultValue = "10") Integer size,  // 使用 size
  ```

**影响**:
前端传递的 `pageSize` 参数无法被后端正确接收，后端会使用默认值10。

**建议修改方案**:
统一使用 `size` 作为参数名：

```javascript
// Books.vue 第98行修改为：
const res = await getBookList({
  page: currentPage.value,
  size: pageSize.value,  // 改为 size

  categoryId: categoryFilter.value,
  status: statusFilter.value,
  sortBy: sortField.value,
  sortOrder: sortOrder.value === 'descending' ? 'desc' : 'asc'

```

### 🔵 问题8：状态筛选重置值为空字符串而非null

**问题描述**:
`Books.vue` 第127-133行的重置搜索函数中，将筛选条件重置为空字符串：

```javascript
function handleReset() {
  searchQuery.value = ''
  categoryFilter.value = ''  // 应该是 null
  statusFilter.value = ''    // 应该是 null
  currentPage.value = 1
  fetchBooks()
}
```

**影响**:
空字符串 `''` 与 `null` 在后端处理时行为可能不同：
- 后端使用 `StringUtils.hasText()` 判断时，空字符串会被视为"有值"
- 可能导致查询条件异常

**建议修改方案**:
```javascript
function handleReset() {
  searchQuery.value = ''
  categoryFilter.value = null  // 改为 null
  statusFilter.value = null    // 改为 null
  currentPage.value = 1
  fetchBooks()
}
```

### 🔵 问题9：缺少自定义业务异常类

**问题描述**:
项目没有定义统一的业务异常类，导致异常处理不够规范。

**影响**:
- 错误码和错误信息管理不规范
- GlobalExceptionHandler需要处理所有RuntimeException
- 无法区分不同类型的业务错误

**建议修改方案**:
参见问题6的建议，创建 `BusinessException` 异常类，并在 `GlobalExceptionHandler` 中添加专门处理：

```java
@ExceptionHandler(BusinessException.class)
public ApiResponse<Void> handleBusinessException(BusinessException e) {
    return ApiResponse.error(e.getCode(), e.getMessage());
}
```

### 🔵 问题10：ApiResponse类缺少Setter方法

**问题描述**:
`ApiResponse.java` 使用了 `@Getter` 但没有 `@Setter`，这是有意设计为不可变对象，但如果需要序列化或某些框架需要Setter，可能会出问题。

**影响**:
当前设计合理，无实际影响。

**建议**:
保持当前设计，Immutable对象是好的实践。如果某些序列化框架需要Setter，可以考虑使用 `@AllArgsConstructor` 配合 `@Getter`。

## 四、优秀实践总结

尽管存在上述问题，代码中也体现了以下优秀实践：

### ✅ 项目规范遵守良好

1. **Entity和DTO正确使用@Getter/@Setter**: 所有Entity类（Admin、Book、Category）和DTO类都使用了 `@Getter/@Setter`，避免了 `@Data` 可能带来的问题。
2. **Service层使用接口+实现模式**: `AdminService`、`BookService` 都有接口定义和对应的实现类，符合分层架构原则。
3. **使用@Resource注入依赖**: 所有Service实现类都使用 `@Resource` 注入Mapper和其他Service，而非 `@Autowired`。
4. **使用LambdaQueryWrapper**: 查询条件构建使用了MyBatis-Plus的 `LambdaQueryWrapper`，避免了字段名硬编码。
5. **统一响应格式**: 所有Controller返回 `ApiResponse<T>` 统一格式，前端处理一致。

### ✅ 前端代码规范

1. **使用Composition API**: Vue组件使用 `<script setup>` 和Composition API，代码简洁。
2. **API模块化**: API请求独立封装在 `admin.js`，便于维护。
3. **路由守卫完善**: 路由配置中正确实现了管理员权限校验。
